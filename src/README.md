## 전반적인 구현 사항

평소 관심있던 주제를 기반으로 과제를 진행하고 싶어 **팀 스포츠 토너먼트 시뮬레이션** 프로그램을 주제로 하였습니다.

자바에서 클래스를 통해 얻을 수 있는 이점이 무엇일지 생각하는 중

**새로운 요구사항**이 추가되어도<br>
기존 클래스를 **상속**하거나, **재사용**함으로 추가적인 코드를 최소화하며 이를 구현하는 것에 있다 생각이 들었습니다.<br>

이에 전체적인 팀 스포츠에서 공통적으로 들어가는 사항을 넣는 `Team`클래스를 만들었고,<br>
그 중 축구 종목의 서비스를 먼저 구현하고자 `Team`클래스를 상속받는 `FootballTeam`클래스를 생각하였습니다.

이어서 이차 상속을 통해, `FootballTeam`클래스의 객체 중 UEFA챔피언스리그 16강에 올라간 팀들을<br>
따로 분류하기 위해 추가로 `FootballTeam`클래스를 상속받는 `UefaTeam`클래스를 만들었습니다.

전체적인 시뮬레이션의 진행은 `controller`클래스를 통해 진행하게 하였고<br>
시뮬레이션 도중 사용자와의 상호작용(입/출력)을 담당하는 부분은<br>
`view`패키지 내 `InputView`, `OutputView`클래스가 각각 책임을 맡도록 하였습니다.

경기 결과에 대한 부분은 `model`패키지 내에 있는 클래스 속 데이터를 바탕으로<br>
로직을 수행하는 클래스인 `MatchService`의 객체를 `controller`의 멤버로 하여 결과를 얻게 하였습니다.

프로그램의 전반적인 진행에 쓰이는 리터럴 상수값들은 `common` 패키지 내 `enum`을 활용하여<br>
비슷한 분류에 따라 열거형을 나누어 관리하도록 하였습니다.

## 아쉬웠던 점

### 1. 의미없는? 상속

프로그램을 완성하고보니 사실상 주기능은 **챔피언스리그 토너먼트 진행** 단 하나이기에<br>
`Team`클래스와 `FootballTeam`, `UefaTeam`클래스들을 상속을 통해 여러 개로 나눈 것의 의미가<br>
해당 프로그램 내에서는 없었던 것 같습니다.

원 설계 목적상 챔피언스리그 토너먼트 이외에도 타 축구 토너먼트 대회나<br>
타 종목에 대해 다루기 위해 상속구조를 나눴던 것인데,<br>
이를 하나의 대회만 구현하여 의미가 퇴색된 점이 아쉬웠습니다.

토너먼트 대회를 프로그램적으로 구상하는 단계에서 너무 많은 시간을 소모하여 다른 대회까지 적용하지 못하였다보니
<br>
막상 하나의 대회만을 다루어 클래스를 사용하는 본질적 의미가 깨진 것 같았습니다.

### 2. MVC 패턴

프로그램을 구상하며 구현할 때는 마냥 익숙했던 `MVC`패턴 구조를 사용하였는데,<br>
완성하고 난 후 다른 구조가 더 좋진 않았을까? 생각이 들었습니다.<br>
사실 아는 디자인 패턴 구조도 MVC밖에 없었기에 사용한 점도 있지만
추가적으로 찾아보며 학습하지 않았던 점에서 스스로 최선을 다 못한 것 아닌가 하는 아쉬움이 남았습니다.

### 3. 리터럴 관리

`enum`을 통해 리터럴을 관리하는 방식에 대해서 기존에 알고 있었기에<br>
이를 코드 구현에 사용하였으나

```java
for (int i = 0; i < ...)
```

위 코드에서의 `0`처럼, 코드를 바라볼 때 직관적으로 이해하기 쉬운 리터럴 부분조차<br>
`Constant.ZERO.getValue()`와 같은 방식으로 처리한게

돌이켜 생각해보니 오히려 더 가독성 측면에서 불편하진 않나 생각이 들었습니다.
